# Фаза 8: Прозрачная логика запуска и выполнения рассылки

## Цель фазы
Сделать поведение рассылки полностью понятным пользователю до нажатия кнопки «Запустить рассылку», и убрать ложное ожидание, что «все посты уйдут сразу».

Важно: в этой фазе только план и UX-описание. Код не пишем.

---

## 1) Зафиксированные решения

### 1.1 Режим постов
- Принято: **ротация 1 пост за 1 запуск**.
- Режим «все посты за один запуск» не добавляем.

### 1.2 Что такое «запуск»
- «Запуск» = один полный прогон отправки в выбранные активные группы с одним текущим постом.
- Ручной запуск: когда пользователь нажал «Подтвердить» в шаге 3.
- Авто-запуск: каждый отдельный слот расписания (например, ПН 10:00) = отдельный запуск.
- Пример: если есть ПН 10:00, то в ПН 10:00 выполняется один запуск: один пост -> все выбранные группы.

### 1.3 Антибан-интервал
- Целевое поведение (TO-BE): плавающий интервал между отправками в группы **от 5 до 10 секунд**, случайно на каждом шаге.
- На уровне настроек пользователя это не выносить.

### 1.4 Краткое предупреждение для пользователя
- Добавить короткий предупреждающий текст перед стартом и в экране «в процессе»:
  - «Посты отправляются по очереди, не мгновенно».
  - «Интервал 5–10 сек между группами для снижения риска блокировок Telegram».
- Цель: пользователь не пугается, что «поста нет», если сразу пошел проверять.

### 1.5 Ссылка на опубликованный пост (`link`)
- В итоге запуска показывать одну кнопку: `📄 Посмотреть публикации`.
- На экране публикаций выводить ссылки в текстовом списке (`@group -> link`), без 20 отдельных inline-кнопок.
- Основа: сохраняем `message_id` отправленного сообщения по каждой группе.
- Для публичных чатов с `@username`: строим URL вида `https://t.me/<username>/<message_id>`.
- Для групп без публичного username: показать «link недоступен для этого чата».

## 2) AS-IS (как работает сейчас)

- Один запуск берет один пост из пула по `rotation_index`.
- Внутри запуска этот пост идет во все выбранные активные группы.
- Отправка последовательная (не параллельная).
- Текущий интервал:
  - массовая: 5.0 + random(0..1.0) сек,
  - тест: 1.5 + random(0..0.5) сек.
- После успешного запуска (есть хотя бы 1 успешная отправка) ротация сдвигается на следующий пост.
- Авто-слоты проверяются фоновым циклом; слот срабатывает в точную минуту `HH:MM`.
- Если бот был офлайн в эту минуту, слот за этот день пропускается.
- После первого успешного массового запуска выставляется `started_at`; до этого авторассылка не идет.
- Остановка/возобновление управляет только авторассылкой, расписание сохраняется.

## 3) UX-план (TO-BE)

### 3.1 Экран «Запуск рассылки» (шаг 3, перед подтверждением)
Показать:
- какой пост уйдет сейчас (номер в пуле),
- сколько групп в этом запуске,
- предупреждение «по очереди, 5–10 сек между группами»,
- какой пост будет следующим после этого запуска.

Короткий блок:
```text
⚠️ Важно
Посты не отправляются сразу во все группы.
Отправка идет по очереди с интервалом 5–10 сек (антибан).
```

### 3.2 Экран «Рассылка в процессе»
- Один обновляемый статус-месседж с прогрессом:
  - «Обработано X из Y групп».
  - «Осталось примерно ~N сек».
- Кнопок на этом экране нет.
- Не спамить отдельными сообщениями на каждый шаг.

### 3.3 Итог запуска
- Одно финальное сообщение:
  - успешно/ошибка/блок,
  - сколько постов списано,
  - следующий пост в ротации.
- Кнопки внизу:
  - `📄 Посмотреть публикации`
  - `◀️ Назад`

### 3.4 Ограничения по `link`
- Если у группы нет публичного `@username`, ссылку не строим.
- Для таких случаев в списке показываем: `@group -> link недоступен (приватный чат)`.

### 3.5 Экран «Публикации»
- Отдельное сообщение со списком:
  - `1) @group_one -> https://t.me/group_one/123`
  - `2) @group_two -> https://t.me/group_two/456`
  - `3) @private_group -> link недоступен (приватный чат)`
- Кнопка: `◀️ Назад`.

## 5) Сценарии проверки (для будущей реализации)

1. 2 поста, 5 групп, 3 запуска подряд:
- запуск 1 -> пост 1 во все 5 групп,
- запуск 2 -> пост 2 во все 5 групп,
- запуск 3 -> снова пост 1.

2. Проверка значения «запуск»:
- ПН 10:00 = один авто-запуск = один пост в выбранные группы.

3. Проверка антибана:
- интервал между группами плавает в диапазоне 5–10 секунд.

4. Проверка `link`:
- в публичных группах ссылка формируется и открывается из текстового списка,
- в приватных/без username отмечается как недоступно.

---

## 6) Критерии готовности фазы 8

- Пользователь явно видит предупреждение, что отправка не мгновенная.
- Термин «запуск» однозначен: один слот/одно нажатие подтверждения = один запуск.
- Ротация 1 пост/запуск зафиксирована в UI и текстах.
- В финале запуска есть одна кнопка `📄 Посмотреть публикации`, а ссылки выводятся текстовым списком.

## 7) Рекомендованные Production Defaults

### 7.1 Планировщик и долгие паузы
- Использовать `tick scheduler` (проверка расписания каждые 20–60 сек), а не `sleep` на часы/дни.
- Рекомендованный дефолт: `tick = 20 сек`.
- Причина: отсутствие зависаний на длинных интервалах (день/неделя между слотами).

### 7.2 Защита от параллельных запусков
- Один запуск на пользователя одновременно (`lock` на кампанию).
- Если пользователь нажал запуск повторно — показать «рассылка уже выполняется».

### 7.3 Heartbeat и защита от зависаний
- Во время выполнения обновлять `last_heartbeat`.
- Если heartbeat не обновлялся дольше порога, считать запуск зависшим и завершать его как `failed/stuck`.
- Рекомендованный дефолт: `heartbeat timeout = 180 сек`.

### 7.4 Retry/Backoff при временных ошибках
- Временные ошибки (таймауты, FloodWait) — retry с backoff.
- Постоянные ошибки (нет прав, не участник) — не ретраить в этом запуске, фиксировать в отчёте.
- FloodWait всегда уважать (пауза на требуемое количество секунд).

### 7.5 Circuit Breaker (утверждено)
- Если подряд происходит 5 неудачных запусков — автоматически ставить авторассылку на паузу.
- Рекомендованный дефолт: `max_consecutive_failed_runs = 5`.
- После автопаузы отправлять уведомление с действием: пройти готовность/тест и возобновить.

### 7.6 Поведение при офлайне
- Если бот был офлайн в точную минуту слота — слот помечать как `missed/skipped`.
- Задним числом рассылку не догонять.

### 7.7 Сохранение состояния (перезапуск бота)
- Сохранять в persistent state:
  - `started_at`,
  - `rotation_index`,
  - `last_runs`,
  - `consecutive_failed_runs`,
  - `last_heartbeat`,
  - итог последнего запуска.
- После рестарта бот должен корректно продолжать по расписанию без потери контекста.

### 7.8 Обновления UI без спама
- Экран «в процессе» обновлять через `edit_message` с троттлингом.
- Рекомендованный дефолт: обновление не чаще 1 раза в 5–10 сек или каждые 3–5 обработанных групп.

### 7.9 Преполет-проверка перед каждым авто-запуском
- Перед запуском слота проверять:
  - аккаунт подключен,
  - есть посты в пуле,
  - есть активные выбранные группы,
  - достаточный баланс.
- При провале — слот `skipped` + понятное уведомление пользователю.
