## 🚀 Три тарифа и ценообразование

### 📦 Структура в Stripe (3 цены на 1 продукт)

```
Продукт: "Посты для рассылки"
├── Цена 1: €3.99 (100 постов) → small_100_posts
├── Цена 2: €7.99 (300 постов) → medium_300_posts  ⭐ основной выбор
└── Цена 3: €33.99 (1500 постов) → large_1500_posts
```

### 💰 Психология цен

| Пакет | Цена | Посты | €/пост | Скидка | Стратегия |
|-------|------|-------|--------|--------|-----------|
| **Small** | €3.99 | 100 | €0.0399 | — | Входная цена (пробующие) |
| **Medium** ⭐ | €7.99 | 300 | €0.0266 | 33% | Выглядит в 2x дешевле (€11.97 vs €7.99) |
| **Large** | €33.99 | 1500 | €0.0226 | 73% | Якорь — дорого, но делает Medium выгодным |

**Ожидаемое распределение:**
- Small: 10% (новички)
- **Medium: 85%** (основная масса) ← целевой пакет
- Large: 5% (профессионалы)

**Средняя выручка:** €8.19/клиент → €409.50 при 50 клиентах/месяц

## 🛠️ Stripe Setup Чеклист

### ✅ Шаг 1: Создать продукт в Stripe

**Stripe Dashboard → Catalog → Add Product**

```
Name: Посты для рассылки
Description: Пакеты постов для автоматической рассылки в Telegram
Type: Standard (один раз, не подписка)
```

### ✅ Шаг 2: Добавить 3 цены

**На странице продукта → + Add price** (повторить 3 раза)

#### Цена 1: Small (100 постов / €3.99)
```
Billing period: One-time
Price: 3.99
Currency: EUR
Nickname: small_100_posts
Lookup key: small_100_posts
→ СКОПИРОВАТЬ Price ID → small_price_id
```

#### Цена 2: Medium (300 постов / €7.99)
```
Billing period: One-time
Price: 7.99
Currency: EUR
Nickname: medium_300_posts
Lookup key: medium_300_posts
→ СКОПИРОВАТЬ Price ID → medium_price_id
```

#### Цена 3: Large (1500 постов / €33.99)
```
Billing period: One-time
Price: 33.99
Currency: EUR
Nickname: large_1500_posts
Lookup key: large_1500_posts
→ СКОПИРОВАТЬ Price ID → large_price_id
```

### ✅ Шаг 3: Получить API ключи

**Stripe Dashboard → Developers → API keys**

```
Secret key (для разработки): sk_test_...
→ СКОПИРОВАТЬ → STRIPE_SECRET_KEY в .env
```

### ✅ Шаг 4: Создать Webhook endpoint

**Stripe Dashboard → Developers → Webhooks → + Add endpoint**

```
Endpoint URL: https://your-railway-app.up.railway.app/stripe-webhook
Events:
  ✓ payment_intent.succeeded
  ✓ charge.refunded

Signing secret: whsec_...
→ СКОПИРОВАТЬ → STRIPE_WEBHOOK_SECRET в .env
```

## 📝 Детали каждой фазы

### **Фаза 1: Баланс постов в JSON + UI** ✅
📌 Файл: [01_phase_balance_system.md](01_phase_balance_system.md)

**Что делается:**
- ✅ `broadcast_state.json` → добавить поле `balance` (dict с `total`, `available`, `history`)
- ✅ Меню "💰 Баланс постов" с текущим балансом
- ✅ Меню "🛍 Купить посты" (3 варианта)
- ✅ При `/start` → дать 30 бесплатных постов
- ✅ Отображение истории: "Потрачено X постов на рассылку Y"

**Зависимости:** None (первая фаза)

**Результат:** UI полностью работает, но кнопки "Купить" еще не обрабатывают платежи

---

### **Фаза 2: Stripe webhook + обновление баланса** 📝
📌 Файл: [02_phase_stripe_integration.md](02_phase_stripe_integration.md)

**Что делается:**
- ✅ Создать `bot/stripe_handler.py` с webhook endpoint
- ✅ Валидация webhook подписи (security)
- ✅ При `payment_intent.succeeded` → обновить баланс пользователя
- ✅ Кнопки "Купить" открывают Stripe Checkout → redirect на success/cancel
- ✅ На success → подтверждение в боте, ссылка на уведомление платежа

**Зависимости:** Фаза 1 (баланс UI должна работать)

**Результат:** Реальные платежи через Stripe, баланс обновляется автоматически по webhook

---

### **Фаза 3: Проверка баланса перед рассылкой** 🔍
📌 Файл: [03_phase_balance_check.md](03_phase_balance_check.md)

**Что делается:**
- ✅ Перед массовой рассылкой: `balance.available >= len(selected_groups)`
- ✅ Перед тестом: проверить баланс, но **не тратить посты** (тест бесплатный!)
- ✅ Меню подтверждения: "Рассылка обойдется в X постов"
- ✅ Если баланса нет → предложить купить

**Зависимости:** Фаза 1, 2 (должен быть баланс и возможность его пополнить)

**Результат:** Без баланса рассылка невозможна, но тест всегда работает

---

### **Фаза 4: Списание постов после успешной рассылки** 💳
📌 Файл: [04_phase_deduction_logic.md](04_phase_deduction_logic.md)

**Что делается:**
- ✅ После `send_broadcast_campaign()` получить `sent_count` (реально отправленные)
- ✅ Списать только успешные: `balance.available -= sent_count`
- ✅ Обновить историю: добавить запись о рассылке
- ✅ Если баланс ≤ 0 → отправить уведомление

**Зависимости:** Фаза 1, 3 (баланс и проверка)

**Результат:** Посты тратятся ТОЛЬКО за реально опубликованные сообщения (не за ошибки)

---

### **Фаза 5: Безопасность тестов (send-check-delete)** 🧪
📌 Файл: [05_phase_test_safety.md](05_phase_test_safety.md)

**Что делается:**
- ✅ Тестовое сообщение отправляется с меткой 🧪
- ✅ Через 60 сек бот проверяет: сообщение еще в группе?
- ✅ Если да → успешно. Если нет → ошибка
- ✅ Бот **сам удаляет** свои тестовые сообщения из групп
- ✅ Пользователь может запускать тесты сколько угодно (не тратит баланс)

**Зависимости:** Фаза 4 (список отправленных сообщений)

**Результат:** Группы остаются чистыми, тест не оставляет мусора

---

### **Фаза 6: Уведомления о низком балансе** 🔔
📌 Файл: [06_phase_balance_notifications.md](06_phase_balance_notifications.md)

**Что делается:**
- ✅ Добавить в `balance` поле `low_balance_threshold` (default 30)
- ✅ В `scheduler_loop()` проверять каждый день: `balance.available <= threshold`
- ✅ Отправить уведомление один раз в день (если условие выполнено)
- ✅ Меню "⚙️ Настройки" → "Пороги уведомлений" (пользователь может изменить 30)

**Зависимости:** Фаза 1, 4 (баланс и его изменения)

**Результат:** Пользователь вовремя узнает когда баланс заканчивается

---

### **Фаза 7: Защита от abuse (rate limiting, cooldown)** 🛡️
📌 Файл: [07_phase_abuse_protection.md](07_phase_abuse_protection.md)

**Что делается:**
- ✅ Cooldown между кликами на "Тест" (30 сек)
- ✅ Лимит на тесты в день (макс 5 в день)
- ✅ Логирование подозрительной активности
- ✅ **OWNER_ID** может запускать тесты неограниченно (разработчик)

**Зависимости:** Фаза 5 (механизм тестов)

**Результат:** Защита от спама и ненужных трат ботом

## 🎯 Поток денег: от платежа до списания

```
1. Пользователь кликает "Купить" (Фаза 1)
                    ↓
2. Открывается Stripe Checkout (Фаза 2)
                    ↓
3. Пользователь платит 💳 (Stripe)
                    ↓
4. Webhook приходит на https://...app/stripe-webhook (Фаза 2)
                    ↓
5. Баланс обновляется: +100/300/1500 постов (Фаза 2)
                    ↓
6. Пользователь настраивает рассылку (Фаза 1)
                    ↓
7. Перед рассылкой проверяется баланс (Фаза 3)
                    ↓
8. Рассылка отправляется (Parser)
                    ↓
9. После рассылки проверяется sent_count (Фаза 4)
                    ↓
10. Баланс списывается: -sent_count постов (Фаза 4)
                    ↓
11. История обновляется (Фаза 4)
                    ↓
12. Если баланс ≤ threshold → уведомление (Фаза 6)
```
