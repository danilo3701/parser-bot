# 📋 PHASES OVERVIEW — MVP Система баланса постов + Stripe

Полная дорожная карта разработки системы оплаты и баланса для бота рассылки.

---

## 🎯 Фазы реализации

### ✅ Фаза 1: Баланс постов в JSON + UI отображения
**Статус:** План готов  
**Файл:** [01_phase_balance_system.md](01_phase_balance_system.md)

**Что делается:**
- Добавить поле `balance` в `broadcast_state.json`
- Меню "💰 Баланс постов" с отображением текущего баланса
- Меню выбора тарифов (4 варианта)
- Инициализация 30 бесплатных постов при /start
- История изменений баланса

**Результат:** Пользователь видит баланс и может выбрать тариф (но кнопки еще не работают)

---

### 📝 Фаза 2: Stripe интеграция (webhook + списание постов)
**Статус:** План в разработке  
**Файл:** [02_phase_stripe_integration.md](02_phase_stripe_integration.md)

**Что делается:**
- Создать Stripe produktы с `price_id` для каждого тарифа
- Реализовать endpoint для Stripe webhook на Railway
- При успешной оплате: обновить баланс через webhook
- Проверка подписи webhook (security)
- Redirect на платеж и обработка success/cancel страниц

**Результат:** Реальные платежи через Stripe, баланс обновляется автоматически

---

### 🔍 Фаза 3: Проверка баланса перед рассылкой
**Статус:** План в разработке  
**Файл:** [03_phase_balance_check.md](03_phase_balance_check.md)

**Что делается:**
- Перед "Массовой рассылкой" проверить: `balance >= len(selected_groups)`
- Перед "Тестом" проверить, но **не тратить посты** (test должен быть бесплатным)
- Показать в меню подтверждения сколько будет потрачено

**Результат:** Без баланса рассылка невозможна, но тест всегда бесплатный

---

### 💳 Фаза 4: Списание постов после успешной рассылки
**Статус:** План в разработке  
**Файл:** [04_phase_deduction_logic.md](04_phase_deduction_logic.md)

**Что делается:**
- После `execute_broadcast()` проверить `sent_count` 
- Списать только количество успешных публикаций
- Обновить `total_spent` в истории
- Если баланс закончился — отправить уведомление

**Результат:** Посты тратятся только за реально опубликованные сообщения

---

### 🧪 Фаза 5: Безопасность тестов (отправка-проверка-удаление)
**Статус:** План в разработке  
**Файл:** [05_phase_test_safety.md](05_phase_test_safety.md)

**Что делается:**
- Тестовое сообщение отправляется с меткой (например, с emoji 🧪)
- Через 60 секунд бот проверяет: сообщение еще в группе?
- Если да → успешно, очищаем результаты. Если нет → не успешно
- После теста бот **сам удаляет свои тестовые сообщения** из групп
- Пользователь может запускать тест сколько угодно раз (не тратит посты)

**Результат:** Группы остаются чистыми, тест безопасен от спама

---

### 🔔 Фаза 6: Уведомления о низком балансе
**Статус:** План в разработке  
**Файл:** [06_phase_balance_notifications.md](06_phase_balance_notifications.md)

**Что делается:**
- Добавить в `balance` поле `low_balance_threshold` (default 30)
- В `scheduler_loop()` проверять: `balance <= threshold`
- Отправить уведомление один раз в день (если баланс низкий)
- Меню "Настройки" → "Пороги уведомлений" (user может изменить 30 на свою цифру)

**Результат:** Пользователь вовремя узнает когда баланс заканчивается

---

### 🛡 Фаза 7: Защита от злоупотребления (rate limiting, cooldown)
**Статус:** План в разработке  
**Файл:** [07_phase_abuse_protection.md](07_phase_abuse_protection.md)

**Что делается:**
- Cooldown между кликами на "Тест" (например, 30 секунд)
- Лимит на количество тестов в день (например, макс 5 в день)
- Логирование подозрительной активности
- OWNER_ID может запускать тесты бесконечно (разработчик)

**Результат:** Защита от спама и растрат ботом
